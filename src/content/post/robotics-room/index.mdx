At Lúnasa, we had a problem: how do you run representative hardware-in-the-loop tests to verify that your computer vision, filters, guidance algorithms and controllers all work together?

To solve this, we built our own lab (quite literally!). Keep an eye out for my LinkedIn post showing the timelapse of us lifting panels and screwing them together.

The result turned out pretty well! Some specs:

| Arm Payload capacity | Up to 30 kg |
| :-- | :-- |
| Degrees of Freedom | 6 DoF arms + extra DoF from the axis |
| Accuracy | 0.05 mm |
| Length of testbed | 7 m |
| Arm reach | Up to 1.3 m |
| Software Compatibility | ROS2, Python |
| Control Frequency | 500 Hz |
| Motion Sensors and Tracking | 6 units with 0.2mm accuracy |

For more details, check out [Cosmic Lab](https://lunasaspace.com/services/) at Lúnasa.

My role was designing the simulator running all the backend — handling all the reference frame transformations from RTN satellite positions to actual robot positions in the room, and... *lots* of quaternion debugging. *Too many* quaternion bugs.

> **Fun fact:** taking into account algebraic definition, storage order, function of rotation, handedness, and basis vector mapping, you can define a quaternion in **384 ways**... And it seems like every person chooses a different convention!

I was also the lead algorithm and software developer for the chaser satellite's flight code — more on that in [RPO Kit Development](/work/rpo-kit-development)!
